# cafe24-doc-update

## 목적

Cafe24 커스터마이징은 문서화 안 된 플랫폼 거동(옵티마이저 번들, UA 스킨 분기, 모듈/템플릿 특성 등)에
자주 부딪힌다. 한 세션에서 힘들게 알아낸 걸 흘려보내면 다음 세션이 같은 걸 또 파야 한다. 이 스킬은
**세션에서 실측으로 확인한 재사용 가능한 카페24 특성**만 골라 해당 CLAUDE.md에 자연스럽게 축적한다.

## 대상 문서 라우팅

두 층위가 있다. 인사이트 성격에 따라 넣을 곳을 고른다.

- **플랫폼 일반 (Cafe24/SmartDesign 전체에 해당)** → `~/dev/works/cafe24/CLAUDE.md` (공용, 영문 톤)
  - 예: 옵티마이저 번들 거동, UA 기반 PC/모바일 스킨 분기, 템플릿 변수·모듈 규칙, `{$...}` 처리 방식.
- **몰 특정 (이 몰의 파일 구조·운영 관례에만 해당)** → `~/dev/works/cafe24/<mall>-cafe24/CLAUDE.md`
  - 예: 특정 상품 렌더러가 이 스킨의 어느 파일에 있다, 이 몰의 커스텀 컴포넌트 동작.

애매하면 사용자에게 어느 문서인지 물어본다. 기본값은 공용 문서.

## 무엇을 담고 무엇을 버리나 (필터)

**담을 것** — 아래를 모두 만족:
- **비자명**: 공식 문서만 봐선 모르고, 세션에서 실측/역추적으로 확인한 것.
- **재사용 가능·durable**: 다음 세션/다른 작업에서도 통하는 일반 규칙. 상대 날짜는 절대 날짜로.
- **실측 근거**: "이렇게 동작하더라"를 관찰로 뒷받침. 기존 공용 문서의 "confirmed by measurement" 톤과 맞춘다.

**버릴 것**:
- 이 대화에서만 의미 있는 일회성 디테일, 특정 버그의 수정 내역(그건 git이 기록).
- 코드 구조·파일 위치처럼 리포지토리를 보면 아는 것(단, "옵티마이저 때문에 파일명이 안 보인다" 같은
  플랫폼 특성은 예외로 담을 가치가 있음).
- 이미 대상 문서에 있는 내용(반드시 dedupe — 있으면 새로 쓰지 말고 기존 항목을 보강).

## 워크플로우

### 1. 대상 문서 읽기
넣을 CLAUDE.md를 **먼저 통독**한다. 섹션 구성(`##` 헤딩), 어투(영문/국문), 상세도, 불릿 스타일을 파악한다.
새 내용은 이 문서의 목소리로 써야 한다 — 튀지 않게.

### 2. 세션에서 후보 추출
현재 대화 맥락을 훑어 위 필터를 통과하는 카페24 특성을 뽑는다. 각 후보에 대해:
- 한 줄 요지(관찰 근거 포함)
- 대상 문서(공용/몰) 및 **어느 기존 섹션에 붙일지 / 새 섹션이 필요한지**
- 기존 문서와 중복 여부

### 3. 사용자에게 확정받기
후보를 **간결한 목록**으로 제시하고 무엇을 반영할지 고르게 한다(전부 넣지 말 것 — 취사선택이 핵심).
`AskUserQuestion`(multiSelect)이나 번호 목록으로. 문구 조정 요청도 받는다.

### 4. 병합 (톤·구조 유지)
선택된 항목을 대상 문서에 반영한다. **관련 기존 섹션이 있으면 거기에 불릿/문장으로 얹고**, 없을 때만
기존과 같은 무게감의 `##` 섹션을 새로 만든다. 원칙:
- 기존 문서의 언어·어투·상세도에 맞춘다(공용 문서면 영문, 측정 기반 서술).
- 추가 블록이 기존보다 더 크거나 화려하지 않게 — 자연스럽게 섞인다.
- `{$...}`·모듈 문법 등 카페24 특수 표기는 보존.
- 간결하게. 장황한 재현 서사 말고, 재사용 가능한 규칙과 근거만.

### 5. 검토용 제시
수정한 부분을 diff로 보여주고, 커밋은 사용자에게 맡긴다(임의로 커밋하지 않음).

## 스타일 예시 (공용 문서 톤)

기존 공용 CLAUDE.md는 이런 식이다 — 이 결에 맞춰라:

```markdown
## HTML Optimizer — @js/@css bundling (behavior confirmed by measurement)

- The order of @js declarations within the same file is preserved in the bundle, but
  the order across files ... There have been cases where ...
```

새 항목도 동일하게: 짧은 관찰형 불릿, 왜 중요한지/어떻게 대응하는지 한 줄, 필요하면 근거.

## 주의
- 대상 문서가 국문이면 국문으로, 영문이면 영문으로 쓴다(현 공용 문서는 영문).
- 애매한 인사이트를 "혹시 몰라" 넣지 말 것 — 확신 있는 것만. 문서가 잡음으로 붇으면 가치가 떨어진다.
- 여러 몰에 걸친 특성인지 한 몰 특성인지 헷갈리면 물어보고, 공용/프로젝트 문서를 정확히 고른다.
