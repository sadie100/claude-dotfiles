# 산출 문서 템플릿

## 상세 문서 (`외부-플러그인-분석.md`)

```markdown
# <몰이름> 자사몰 외부 플러그인 분석

- 분석 대상: <스킨 경로들> — N개 파일 전수 스캔
- 분석일: YYYY-MM-DD
- 몰 계열: <도메인들>

## 읽기 전 전제

1. 스킨 소스에 로더가 없는 플러그인이 있다 (관리자 마케팅 스크립트/앱스크립트 주입 — "관리자 주입"으로 표기).
2. <캐시 스냅샷 파일이 있으면> *_cache.html은 스킨 소스가 아니라 렌더 스냅샷 — 주입 ID 역추적 출처로만 사용.
3. 백업/구버전 파일에만 있는 코드는 "미사용(레거시)".
4. <플랫폼 특이사항 — 예: PC 스킨에 앱 모듈 호출 없음>
5. 활성 위치는 라이브 페이지 URL 기준 (PC = <도메인>, MO = <도메인>, 스킨 경로 = URL 경로 — 실측 확인).

## 1. 기능성 서드파티 SaaS
### <서비스명> — <한줄 정체>
- **용도**: (코드 근거 기준)
- **연동방식**: (SDK/iframe/모듈 + 파일 경로:라인)
- **활성 위치/조건**: (라이브 URL 기준, PC/MO 구분, 조건부면 조건)
- **관련 파일**: (경로:라인 나열)
- **계정**: (키/ID/몰코드)

## 2. 이용 중인 카페24 앱
(연동 3유형 설명: 디자인 파일 설치형 / 앱스크립트 주입형 / 백오피스·API형)
### 2-a. 스킨 코드에서 확인된 앱
### 2-b. 관리자 마이앱 대조 결과 (총 N개 — 사용중/만료/삭제, 성격별 분류)
### 2-c. 스킨 잔존 코드 ↔ 관리자 상태 대조 (| 스킨 흔적 | 대응 앱 | 관리자 상태 | 판단 | 표)

## 3. 마케팅 및 트래킹 코드
(| 서비스 | 계정/ID | PC | MO | 요약 표 → 서비스별 상세 → 미사용 레거시 목록)

## 4. 카페24 외 커스텀 데이터
(데이터 레이어 구조와 관리 주체·데이터 흐름 설명 후 항목별 상세.
각 항목에 **관리 메뉴** 필드 인라인 — "없음"이면 운영 주체 직접 요청 영역임을 명시)

## 부록: 수정 시 진입점 요약 (| 기능 | 진입 파일 | 표)
## 부록: 한계 및 후속 확인 필요 사항
```

## 요약 문서 (`외부-플러그인-요약.md`)

```markdown
# <몰이름> 자사몰 외부 플러그인 요약

> 상세 근거·연동방식·파일 위치는 [외부-플러그인-분석.md](./외부-플러그인-분석.md) 참고. PC = <도메인>, MO = <도메인>. (YYYY-MM-DD 기준)

## 1. 기능성 플러그인 및 스크립트
- **서비스명 (사용 페이지)** — 한 줄 설명
- **서비스명 (미사용)** — 설명 + 잔재 사유

## 2. 카페24 앱
(§1과 겹치는 서비스는 §1에만 기재하고 여기선 제외 — "앱으로도 설치돼 있음"만 §2 인트로에 한 줄.
사용중은 성격별 하위 섹션: 프론트 노출형 / 마케팅·전환 / 판매채널·기본설치 / 백오피스·결제 / 자체 프라이빗 앱.
만료·삭제 앱은 현재 미사용 — 전체 명단은 상세 문서에만 두고, 요약엔 "스킨에 코드가 남은 것"만 잔재 섹션으로 표기)

## 3. 분석 및 추적 툴
- **서비스명 (활성 페이지·이벤트)** — 한 줄 설명
### 미사용 잔재 (코드만 남음)

## 4. 커스텀 데이터 (관리: <관리자 URL>)
- **소스명 (사용 페이지)** — 한 줄 설명
```

요약 작성 규칙:
- 항목은 전부 개별 나열 — "등", "다수", "N종"으로 뭉개지 않는다.
- 비개발자 독자 가정 — 기술 용어는 풀어 쓴다 (프록시 → "대신 받아 전달하는 중계 서버").
- 슬랙에 바로 붙일 수 있는 밀도 유지 (한 항목 = 한 줄).
