# Cafe24 외부 플러그인 분석

스킨 코드만 봐서는 전체가 안 보인다는 것이 이 스킬의 출발점이다. 코드 스캔 → 관리자 실측 대조 → 라이브 검증의 3중 확인을 거쳐, 각 항목을 **활성 / 관리자 주입(스킨 무흔적) / 잔재(만료·삭제)** 중 하나로 판정한다.

## 분류 기준 (4분류)

1. **기능성 서드파티 SaaS** — 리뷰·상담·업셀·제휴 등 기능 제공 서비스 (예: 크리마, 채널톡)
2. **이용중인 카페24 앱** — 앱스토어 설치 앱 (예: 배너 앱)
3. **마케팅 및 트래킹 코드** — 픽셀·로그·전환 (예: GA, 네이버 로그분석). **1에 속해도 마케팅/트래킹 성격이면 3으로 분류**
4. **카페24 외 커스텀 데이터** — 외주사/자체 인프라가 데이터를 공급하는 화면 (API, 정적 JSON 등)

## 진행 단계

### 0. 입력 확인

- 스킨 코드 경로(PC/MO 각각)를 확정한다.
- **몰 라이브 도메인은 로컬 설정에서 추론하지 말고 사용자에게 확인**한다 (PC/MO 도메인 각각).
- 카페24 관리자(마이앱), 외주사 관리자 접근 가능 여부를 물어본다 — 없으면 해당 단계는 "코드 기준 한계"로 문서에 명시하고 건너뛴다.
- 완료 기준: 경로·도메인·관리자 접근성 3가지가 확정됨.

### 1. 외부 도메인 전수 스캔

스킨 전체에서 외부 도메인을 빈도순으로 추출해 후보 목록을 만든다:

```bash
grep -rhoE '(https?:)?//[a-zA-Z0-9.-]+\.[a-z]{2,}[^"'\'' )<>]*' --include='*.html' --include='*.js' . \
  | sed -E 's|^(https?:)?//||' | cut -d/ -f1 | sort | uniq -c | sort -rn
```

- 노이즈(w3.org, jquery, console.log 파편 등)를 걸러내고, `references/known-services.md`의 도메인→서비스 표와 대조해 1차 후보를 확정한다.
- 표에 없는 도메인은 버리지 말고 "미확인" 후보로 유지한다.
- 완료 기준: 노이즈 제외 모든 외부 도메인이 서비스 후보 또는 미확인으로 분류됨.

### 2. 4분류 병렬 심층 조사

Explore 서브에이전트 4개를 병렬로 띄워 분류별로 조사한다. 각 브리프에 반드시 포함:

- 1단계에서 찾은 해당 분류의 후보 도메인 목록
- 항목별 수집 필드: 서비스명 / 용도(코드 근거) / 연동방식+파일경로 / 활성 위치·조건(PC·MO 각각) / 계정·키 식별자
- **백업 파일 규칙**: `*copy*`, `*_old*`, `*_bak*`, 날짜 suffix 파일에만 있으면 "백업에만 존재(미사용 추정)"로 구분. 활성 레이아웃 판별 필수 — 파일이 존재해도 어떤 활성 레이아웃도 import하지 않으면 미사용 (예: footer.html이 있어도 활성은 renew_footer.html일 수 있음)
- **캐시 스냅샷 주의**: 라이브 렌더 결과를 통째로 저장한 `*_cache.html` 류는 스킨 소스가 아니다. 관리자 주입 태그의 계정 ID를 역추적하는 출처로만 쓰고, "활성 스킨 코드"로 취급 금지
- 카페24 앱 조사 시: `/apps/` 폴더 + `module="앱명_display_N"` 역추적. **카페24 표준 PG(이니시스/KCP/PAYCO)·배송조회 링크는 플러그인이 아님**
- 커스텀 데이터 조사 시: `fetch|\.ajax|axios|XMLHttpRequest|getJSON` 전수 grep, 카페24 자체 API(`/exec/front`)는 표준으로 구분만
- 완료 기준: 4개 에이전트 결과가 전부 수집되고, 1단계 후보 도메인이 빠짐없이 어딘가에 배정됨.

### 3. 카페24 관리자 마이앱 실측 대조 (접근 가능 시)

스킨의 `/apps` 폴더는 **디자인 파일 설치형(구형) 앱만** 잡는다. 앱스크립트(ScriptTags) 주입형은 스킨에 무흔적, 백오피스/API형은 프론트에 아예 안 나타난다. 그래서 관리자 대조가 필수다.

- 공유 크롬으로 `https://<몰>.cafe24.com/disp/admin/shop1/myapps/list` 접속, `li.item`의 클래스로 상태 추출:

```js
[...document.querySelectorAll('li.item')].map(c => ({
  status: c.classList.contains('using') ? 'using' : c.classList.contains('expired') ? 'expired' : 'deleted',
  title: c.innerText.split('\n').map(s=>s.trim()).filter(Boolean)[0]
}))
```

- (사용중 목록엔 즐겨찾기 중복이 섞이므로 dedup)
- **잔재 판정**: 스킨 흔적 ↔ 관리자 상태 대조표를 만든다. 스킨에 코드가 있어도 앱이 만료/삭제면 "잔재", 사용중인데 스킨에 무흔적이면 "앱스크립트 주입형". §2 결과 중 이 대조로 뒤집히는 항목(웹푸시 서비스워커만 남은 경우 등)을 정정한다.
- 완료 기준: 사용중/만료/삭제 전체 목록 확보 + 스킨 발견 항목 전부에 관리자 상태가 매겨짐.

### 4. 라이브 URL 검증

- 활성 위치는 **라이브 페이지 URL 기준**으로 적는다(코드 경로는 관련 파일 필드로). Cafe24는 스킨 파일 경로가 그대로 URL 경로가 된다(`sell.html` → `<도메인>/sell.html`) — 대표 경로 몇 개를 curl로 실측(200/404)해 매핑 규칙을 확정한다.
- 커스텀 정적 데이터(`/json/*.json` 류)는 레포에 없어도 라이브에 존재할 수 있다 — 전수 curl로 존재 확인.
- 재검증 시 캐시 주의: 같은 URL이 구/신 응답을 섞어 주므로 fresh query param(`?x=<timestamp>`)을 붙인다.
- 완료 기준: 문서에 적을 모든 활성 위치 URL 패턴이 실측으로 확인됨.

### 5. 외주사/자체 관리자 대조 (접근 가능 시)

커스텀 데이터 레이어에 관리자(CMS)가 있다면 메뉴를 전수 추출해 프론트와 1:1 매핑한다.

- 관리자 메뉴(링크 텍스트+href) 추출 → 각 메뉴가 생성하는 데이터(JSON/API)와 소비 화면을 연결
- 각 커스텀 데이터 항목에 **관리 메뉴** 필드를 인라인으로 붙인다(별도 매핑 표로 떨어뜨리면 헷갈린다는 피드백 있었음). "관리 메뉴: 없음" = 관리자 UI 없이 운영되는 인프라 = 수정하려면 운영 주체에 직접 요청해야 하는 영역임을 명시
- 대응 메뉴가 없는 데이터, 스킨에 소비처가 없는 메뉴 둘 다 "미확인"으로 남긴다
- 완료 기준: 모든 커스텀 데이터 항목에 관리 메뉴 필드가 있음(없으면 "없음"으로).

### 6. 문서 2종 산출

`<몰>/_docs/`(또는 사용자 지정 위치)에 상세·요약 문서를 만든다. 포맷은 `references/doc-templates.md`를 따른다.

- **상세 문서**: 읽기 전 전제(관리자 주입/캐시 스냅샷/백업 규칙/URL 규칙) → 4분류 각 항목 상세 → 부록(수정 진입점, 한계·후속 확인)
- **요약 문서**: `- **이름 (사용 페이지)** — 한 줄 설명` 포맷, 4개 섹션. **"등·다수"로 뭉개지 말고 서비스별 전부 개별 나열**. 미사용/잔재는 섹션 끝에 몰아서 표기
- 비개발자 독자를 가정: 프록시→"중계 서버"처럼 기술 용어는 풀어 쓴다
- 완료 기준: 두 문서 모두 존재하고, 요약의 모든 항목이 상세에 근거를 가짐. 확인 못 한 영역(관리자 마케팅 스크립트 직접 삽입분 등)이 한계 섹션에 명시됨.
