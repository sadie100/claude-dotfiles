# dev-process

4단계 개발 프로세스를 순차적으로 실행한다.
각 단계에서 AI가 초안을 작성하고, 사용자가 검증하여 확정한 뒤 다음 단계로 진행한다.

## 프로세스 개요

```
[1. PRD/디자인] → [2. 아키텍처] → [3. 구현 스펙] → [4. 테스트 케이스] → 개발 시작!
   /dev-prd        /dev-architecture   /dev-spec       /dev-testcase
```

| 단계 | AI 역할 | 사용자 역할 | 산출물 |
|------|---------|------------|--------|
| 1. PRD | 초안 작성, 일관성 검토 | 우선순위, 요구사항 결정 | docs/dev-process/PRD.md |
| 2. 아키텍처 | 기술 옵션, 트레이드오프 분석 | 기술 판단, 시스템 영향 평가 | docs/dev-process/ARCHITECTURE.md |
| 3. 구현 스펙 | 상세 스펙 초안, 인수 조건 | 도메인 규칙, 엣지 케이스 검증 | docs/dev-process/SPEC.md |
| 4. 테스트 케이스 | TC/시나리오 자동 도출 | 갭 확인, AC 매핑 검증 | docs/dev-process/TESTCASE.md |

## Workflow

### 1. 기능명 확인

인자로 전달된 기능명을 확인한다. 없으면 물어본다.

출력 디렉토리 `docs/dev-process/`가 이미 존재하고 문서가 있으면 사용자에게 알린다:
- 기존 문서를 덮어쓸지
- 특정 단계부터 재시작할지

### 2. Stage 1 — PRD/디자인

`/dev-prd {기능명}`을 실행한다.

완료 후 사용자에게 확인:
> "PRD가 작성되었습니다. 다음 단계(아키텍처 설계)로 진행할까요?"

사용자가 확인하면 다음 단계로 진행한다.

### 3. Stage 2 — 아키텍처

`/dev-architecture {기능명}`을 실행한다.

완료 후 사용자에게 확인:
> "아키텍처 문서가 작성되었습니다. 다음 단계(구현 스펙)로 진행할까요?"

### 4. Stage 3 — 구현 스펙

`/dev-spec {기능명}`을 실행한다.

완료 후 사용자에게 확인:
> "구현 스펙이 작성되었습니다. 다음 단계(테스트 케이스)로 진행할까요?"

### 5. Stage 4 — 테스트 케이스

`/dev-testcase {기능명}`을 실행한다.

### 6. 완료 리포트

전체 프로세스 완료 후 요약을 보여준다:

```
## 개발 프로세스 완료

**기능명**: {기능명}

### 산출물
| 단계 | 파일 | 상태 |
|------|------|------|
| PRD | docs/dev-process/PRD.md | Confirmed |
| 아키텍처 | docs/dev-process/ARCHITECTURE.md | Decided |
| 구현 스펙 | docs/dev-process/SPEC.md | Confirmed |
| 테스트 케이스 | docs/dev-process/TESTCASE.md | Confirmed |

### 다음 단계
위 문서를 기반으로 구현을 시작할 수 있습니다.
구현 순서는 SPEC.md의 "구현 단위 목록"을 참고하세요.
```

## Notes

- 각 단계는 독립적으로도 실행 가능하다 (`/dev-prd`, `/dev-architecture` 등)
- 중간 단계부터 시작할 수 있다 — 선행 문서가 없으면 해당 스킬이 직접 정보를 수집한다
- 각 단계 사이에 사용자 확인을 받아 불필요한 진행을 방지한다
- 기존 문서가 있으면 덮어쓰기 전에 반드시 사용자에게 확인한다
