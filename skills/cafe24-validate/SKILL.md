# cafe24-validate — Cafe24 페이지 순회 검증

Cafe24 스킨에 변경을 배포한 뒤, 실제 페이지들을 브라우저로 돌면서 "다 잘 나오는지"를 확인하는 스킬이다. 코드 레벨 상태 점검(evaluate)과 육안 확인(스크린샷)을 병행하고, 결과를 상품/페이지별 표로 보고한다.

## 0. 인자 검증 — URL 없으면 즉시 실패 (가장 중요)

이 스킬은 **점검 대상 base URL을 인자로 반드시 받아야 한다.**

- 인자에 URL이 없으면: **아무 점검도 시작하지 말고 즉시 실패로 종료**한다. "점검할 URL을 인자로 주세요 (예: `/cafe24-validate https://몰주소/skin-skin42/ 상세페이지`)"라고 안내하고 끝낸다.
- 기억, 이전 대화, 추측으로 URL을 만들어내지 않는다. **Cafe24 테스트 스킨 번호(skin-skin42, skin-mobile30 등)는 스킨 복사/교체 때마다 바뀐다.** 과거에 쓰던 URL이 지금은 다른 스킨을 가리킬 수 있고, 잘못된 스킨을 점검하면 "정상인데 깨졌다" 또는 "깨졌는데 정상이다"라는 완전히 틀린 보고가 나간다. 이 스킬이 URL을 강제로 받는 이유가 정확히 이 사고를 막기 위해서다.
- URL 외에 **점검 범위**(상세페이지 순회 / 목록페이지 / 메인 / 특정 상품번호들)도 인자로 받는 것이 이상적이다. 범위가 없으면 AskUserQuestion으로 물어본다 (기본 제안: 상품 상세 순회).

시작하기 전에 URL이 실제로 유효한지 확인한다:

```bash
curl -s -o /dev/null -w "%{http_code} %{redirect_url}\n" "<base-url>"
```

200이 아니거나 예상 밖 도메인으로 리다이렉트되면 사용자에게 URL을 재확인받는다 (예: 모바일 스킨 URL에 데스크톱 UA로 접근하면 리다이렉트될 수 있음).

## 1. 준비

- 브라우저는 **Playwright MCP** (`browser_navigate` / `browser_evaluate` / `browser_take_screenshot` / `browser_resize`)를 쓴다. Chrome DevTools MCP는 프로필 충돌로 실패하는 경우가 있으니 폴백으로만.
- **모바일 스킨**(URL에 `skin-mobile` 포함이거나 사용자가 모바일이라고 함)이면 `browser_resize`로 390×844 뷰포트로 맞추고 점검한다. PC 스킨은 기본 뷰포트.
- 스크린샷은 **scratchpad 디렉토리**에 저장한다. 워킹 디렉토리(리포지토리)에 저장하면 저장소가 오염된다 — Playwright 기본 저장 위치가 cwd이므로 확인 후 정리할 것.

## 1.5. 로그인 확인 — base URL이 로그인을 요구하면 즉시 중단

base URL에 처음 navigate한 직후, 로그인이 필요한 상태인지 확인한다:

- 현재 URL이 로그인 페이지로 리다이렉트됐는가 (`/member/login`, `/intro/` 포함 등)
- 페이지에 로그인 폼만 나오거나 "접근 권한이 없습니다" / "접근할 수 없는 등급입니다" 류의 알림·다이얼로그가 뜨는가

로그인이 필요하면 **거기서 순회를 즉시 중단한다.** 우회하거나(다른 URL 시도, UA 변경 등) 자동 로그인을 시도하지 않는다. 대신:

1. 브라우저를 닫지 말고 로그인 페이지가 열린 상태로 둔다.
2. 사용자에게 이렇게 안내하고 턴을 종료한다: "테스트 URL이 로그인을 요구합니다. Playwright가 띄운 브라우저 창에서 직접 로그인해 주세요. Playwright MCP는 persistent 프로필을 쓰므로 한 번 로그인하면 이후 세션에도 유지됩니다. 로그인 후 '계속해줘'라고 알려주시면 순회를 재개합니다."
3. 사용자가 로그인 완료를 알려오면 base URL로 다시 navigate해서 로그인 상태를 확인한 뒤 순회를 시작한다.

이건 base URL 자체가 막힌 경우다. 순회 도중 **개별 상품**만 등급 제한으로 안 열리는 경우는 §3(d)대로 건너뛰고 보고에 명시한다.

## 2. 순회 대상 수집

사용자가 상품 번호들을 지정했으면 그대로 쓴다. 지정하지 않았으면:

1. base URL(메인 또는 목록)을 열고 상품 링크를 수집한다:

```js
() => Array.from(new Set(
  Array.from(document.querySelectorAll('a[href*="product_no="]'))
    .map(a => a.getAttribute("href"))
)).slice(0, 30)
```

2. 수집된 링크에서 **서로 다른 유형의 상품 5~8개**를 고른다. 같은 유형만 여러 개 돌면 커버리지가 없다 — 옵션 구조가 다른 상품(단일옵션/다중옵션/세트), 신상품, 베스트 등을 섞는다. 사이트 전용 체크(§4)가 있으면 그 기준(예: 커스텀옵션 상품 vs 일반 상품)으로 반드시 양쪽을 포함시킨다.
3. 수집한 링크의 도메인이 base URL과 다르면(예: 라이브 도메인 절대경로) base URL 기준 상대경로로 바꿔 접속한다 — 테스트 스킨을 점검하러 왔는데 라이브 도메인으로 새면 안 된다.

## 3. 페이지별 공통 점검

각 페이지에서 navigate 후:

**(a) 코드 레벨 상태** — `browser_evaluate`로 한 번에 수집:

```js
() => ({
  title: document.title,
  // 옵션/구매 영역이 존재하고 보이는가 (사이트 구조에 맞는 셀렉터로 교체)
  optionAreaVisible: (function(){ var el = document.querySelector("table[module='product_option'], .xans-product-option"); return el ? el.offsetParent !== null || getComputedStyle(el).position === "absolute" : false; })(),
  // 가로 오버플로(모바일 레이아웃 깨짐의 대표 신호)
  horizontalOverflow: document.documentElement.scrollWidth > document.documentElement.clientWidth + 1,
  bodyEmpty: document.body.innerText.trim().length < 50
})
```

**(b) 콘솔 에러** — `browser_console_messages`(level: error)로 수집한다. 판단할 때 주의:

- 트래킹/마케팅 스크립트의 에러(`ERR_UNKNOWN_URL_SCHEME`, 서드파티 도메인 로드 실패)와 스킨 자체의 기존 에러(예: 템플릿 변수 미치환 `$xxx is not defined`)는 대부분 **이번 배포와 무관한 기존 문제**다. 첫 페이지에서 나온 에러 목록을 기준선으로 잡고, 이후 페이지에서 **새로 등장하는 에러**에 집중한다.
- 기존 에러도 보고서에는 "기존 문제(이번 변경과 무관)"로 한 줄 언급한다 — 숨기지 말 것.

**(c) 육안 확인** — 전 페이지를 다 찍을 필요는 없고, **유형별 대표 1~2개**는 반드시 `browser_take_screenshot`으로 찍고 **Read 도구로 이미지를 직접 열어서 눈으로 확인**한다. 코드 레벨 상태가 전부 정상이어도 CSS가 깨져 보일 수 있다 — 스크린샷을 찍기만 하고 안 보는 것은 점검이 아니다. 옵션이 모달 안에 있는 스킨(모바일)은 모달을 여는 버튼을 클릭한 뒤 찍는다.

**(d) 접근 불가 상품** — 회원 등급 제한 상품은 비로그인 브라우저에서 "접근할 수 없는 등급입니다" 알림과 함께 열리지 않는다. 이런 상품은 건너뛰고 보고서에 "로그인 필요, 사용자 직접 확인 요망"으로 명시한다.

## 4. 사이트 전용 체크

점검 대상이 아래 사이트면 해당 레퍼런스를 읽고 공통 점검에 **추가로** 수행한다:

| 사이트 | 레퍼런스 | 내용 |
| --- | --- | --- |
| 클룹 (cloop) | `references/cloop.md` | 골라담기(커스텀옵션) 방어 로직 상태 점검 — ready 플래그, 잠금/오버레이, product_submit 래핑, 검증 함수, PC/모바일 템플릿 배포 조합 확인 |

새 자사몰에서 반복 점검이 생기면 같은 형식으로 `references/<사이트>.md`를 추가한다.

## 5. 보고 형식

순회가 끝나면 반드시 다음 구조로 보고한다:

```markdown
## 순회 결과 — <base URL> (<PC/모바일>)

| 페이지/상품 | 유형 | 결과 |
|---|---|---|
| 230 애사비소다 12종 | 골라담기 | ✅ (핵심 상태 요약) |
| 400 원더팝 | 일반 | ✅ 개입 0, 옵션 정상 노출 |
| 494 테스트상품 | - | ⏭️ 등급 제한, 직접 확인 필요 |

- 콘솔 에러: (신규 없음 / 기존 문제 N건 — 무엇인지)
- 육안 확인: (어떤 페이지를 찍어서 봤고 어땠는지)
- 이상 발견 시: 증상 → 확인한 근거 → 추정 원인 순으로 별도 서술
```

이상을 발견하면 **순회를 계속하기 전에 먼저 사용자에게 보고**한다 — 배포 실수(파일이 잘못 올라감)라면 나머지 순회 결과가 전부 무의미해지기 때문이다. 어떤 파일이 잘못 배포됐는지까지 좁힐 수 있으면 좁혀서 알려준다 (레퍼런스 파일에 판별 마커가 있으면 활용).

## 마무리

- 리포지토리에 생긴 스크린샷/임시 파일을 정리한다 (`git status`로 확인).
- 점검은 읽기 전용이다 — 이 스킬에서 코드 수정이나 커밋을 하지 않는다. 발견된 문제의 수정은 별도 작업으로 제안만 한다.
