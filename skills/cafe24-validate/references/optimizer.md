# Cafe24 HTML 옵티마이저 — @js/@css 번들링 (이 스킬이 잡는 사고의 원리)

> 원본·정본은 `dev/works/cafe24/CLAUDE.md`. 이 파일은 글로벌 스킬이 repo CLAUDE.md 없이도 동작하도록 둔 요약본이다. 상충하면 프로젝트 CLAUDE.md가 우선.

Cafe24는 페이지를 서버 렌더(레이아웃 병합 → 치환변수/모듈 해석)한 뒤, HTML 최적화가 켜진 몰에서는 문서 곳곳의 `<!--@js(...)-->`/`<!--@css(...)-->`를 수거해 번들 몇 개로 합쳐 문서 하단에서 로드한다.

- **같은 파일 안 @js 순서는 유지되지만, 파일 간(레이아웃 head vs 페이지 body) 순서는 HTML 문서 순서와 무관하다.** 레이아웃 head 선언 스크립트가 페이지 body 선언보다 뒤에 실행된 사례 있음.
- 같은 파일을 레이아웃·페이지 양쪽에 선언하면 dedupe되며 **페이지 쪽(앞) 자리가 우선**. 순서 보장이 급하면 소비하는 페이지에 중복 선언이 통한다.
- 근본 안전 패턴: **전역값을 로드 시점에 캡처하지 말고 사용 시점에 window에서 읽기(lazy).** `var CONFIG = window.X || {}` 같은 즉시 캡처는 include 순서가 바뀌면 빈 값으로 굳는다.
- 존재하지 않는 파일의 @js는 **에러 없이 번들에서 조용히 빠진다** — 기능이 통째로 안 나오면 파일 404부터 의심.

## 왜 이게 이 스킬의 존재 이유인가

import 순서 역전으로 빈 값이 캡처되면 JS가 **throw 없이 조용히 실패**한다. 결과:

- **콘솔 에러 없음** → 콘솔 모니터링으로 못 잡음.
- **DOM(HTML) 구조 동일** (스타일만 안 먹음) → DOM diff로 못 잡음.
- 그 배포에선 **결정론적으로 깨진 상태** → 스크린샷 1장으로 재현됨.

→ 손 안 댄(unworked) 페이지를 **라이브와 스크린샷 비교**하는 것만이 이 클래스를 탐지한다.

## 검증 시 주의

- 페이지 소스에서 스크립트 파일명 검색해도 안 나오는 게 정상 — 개별 `<script src>`가 아니라 번들 안에 합쳐져 있다. 번들 응답 내용에서 코드 존재·offset으로 확인.
- 번들에 **서버 캐시**가 있어 배포 직후 옛 내용이 나올 수 있다. `filename=`/`t=`가 바뀌어도 내용이 옛것일 수 있으니 시간을 두고 재확인 후 판정.
- **전역 설정값이 콘솔엔 찍히는데 UI가 안 나오면** "로드 안 됨"이 아니라 "실행 순서 역전으로 빈 값 캡처"를 의심.
