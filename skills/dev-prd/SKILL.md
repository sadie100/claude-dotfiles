# dev-prd

기능 개발의 첫 단계로, 제품 요구사항 및 화면/흐름 설계 문서를 작성한다.
AI가 초안을 작성하고, 사용자가 우선순위와 최종 요구사항을 결정한다.

## Workflow

### 1. 기능명 확인 및 컨텍스트 수집

인자로 전달된 기능명을 확인한다. 인자가 없으면 사용자에게 어떤 기능을 기획할지 물어본다.

기존 코드베이스가 있으면 Glob + Read로 프로젝트 구조, README, 기존 문서(docs/ 등)를 탐색하여 컨텍스트를 파악한다.

### 2. 정보 수집 — 1차 질문 (배경/목적)

사용자에게 다음을 질문한다:

- 이 기능의 목적과 해결하려는 문제는 무엇인가?
- 주요 사용자(타겟)는 누구인가?
- 이 기능의 성공 기준(KPI 또는 정성적 목표)은?

### 3. 정보 수집 — 2차 질문 (범위/흐름)

1차 답변을 바탕으로 추가 질문한다:

- 핵심 사용자 시나리오(happy path)를 설명해 달라
- 포함할 기능과 명확히 제외할 기능(scope)은?
- 관련된 기존 기능이나 시스템이 있는가?

### 4. 정보 수집 — 3차 질문 (화면/흐름, 선택)

화면이 관련된 기능이라면 추가로:

- 주요 화면(페이지/뷰)은 몇 개이며 각각의 역할은?
- 화면 간 이동 흐름은 어떻게 되는가?
- 참고할 디자인/와이어프레임/레퍼런스가 있는가?

화면이 없는 기능(백엔드, 배치 등)이면 이 단계를 건너뛴다.

### 5. PRD 초안 작성

수집된 정보를 바탕으로 [references/prd-template.md](references/prd-template.md) 형식에 맞춰 PRD 초안을 작성한다.

작성 원칙:
- 기술 구현이 아닌 **무엇을/왜**에 집중한다
- 사용자 시나리오는 구체적인 step-by-step으로 기술한다
- 범위 밖(Out of Scope) 항목을 명시적으로 적는다
- 우선순위가 불분명한 항목은 [결정 필요]로 표시한다

### 6. 리뷰 및 확정

초안을 사용자에게 보여주고 피드백을 받는다:
- [결정 필요] 항목에 대한 사용자 판단 요청
- 누락된 요구사항 확인
- 우선순위 조정

피드백을 반영하여 문서를 수정한다. 사용자가 확인하면 최종본으로 저장한다.

### 7. Output

`docs/dev-process/PRD.md`에 저장한다.
파일 저장 후 한 줄 요약과 함께 다음 단계(`/dev-architecture`)를 안내한다.
