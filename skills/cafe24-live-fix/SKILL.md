# cafe24-live-fix (FTP 없는 몰: 역추적 → 라이브 검증 → 코드 전달)

## Overview

FTP 미연동 몰은 로컬 미러가 없지만, **원본 스킨 파일은 어차피 웹에서 개별 URL로 서빙되므로 코드 확보에 FTP가 필요 없다.** 이 스킬은 원본을 역추적해 가져오고, 브라우저 주입으로 수정을 확정한 뒤, **수정이 반영된 파일 전문(全文)을 사용자에게 전달하는 것으로 끝난다.** 실제 반영(관리자 스마트디자인 편집창에 붙여넣기)은 사용자가 한다.

**하드 경계 — 이 스킬은 몰의 서버 상태를 절대 바꾸지 않는다.**
- 관리자 페이지 접속·편집창 저장을 하지 않는다. **"제가 대신 반영해 드릴까요?"라는 제안도 하지 않는다** (사용자가 명시적으로 원치 않은 경계).
- 브라우저 주입은 내 탭에만 존재하는 미리보기다. "반영된 거냐"는 질문에는 **"아직 미반영, 전달한 코드를 편집창에 붙여넣어야 반영"**이라고 명확히 답한다.

## Inputs

1. **몰 URL** (라이브 또는 테스트 스킨) — 필수. **반드시 사용자 메시지나 전달받은 프롬프트/티켓에서 받는다.** 없으면 사용자에게 물어본다 — 로컬 디렉토리명·sftp.json 등 설정 파일에서 추론해 계산하지 않는다 (실제 발생: 다른 몰/옛 설정을 집을 수 있다)
2. **요구사항** — 셀렉터/JS 심볼, 화면 요소 설명, 또는 Figma 노드 링크
3. **대상 뷰포트** — 명시 없으면 데스크탑(1280px)

## Workflow

1. **원본 역추적 + 전문 확보.** **REQUIRED SUB-SKILL:** cafe24-find-source 로 원인 파일 경로:라인을 특정하고, 그 파일 **전문을 raw fetch로 받아 스크래치패드에 저장**한다. diff만 들고 진행하지 말 것 — 최종 산출물이 "전문"이므로 원본 전문이 처음부터 필요하다. (몰이 개별 파일 직링크를 막았으면 번들 본문에서 해당 파일 구간을 복원한다.)
2. **주입 검증 루프.** **REQUIRED SUB-SKILL:** cafe24-update 의 4~5단계(body 끝 `<style>` 주입, CSSOM `deleteRule`, 매칭 규칙 스캔)를 그대로 쓴다. **주입 검증은 임시방편이 아니라 이 스킬의 유일한 검증 수단이다** — "배포해서 확인하자"로 건너뛰지 말 것. FTP가 없으니 배포 왕복은 사용자 수작업 1회당 붙여넣기 1번이며, 검증 없이 반복시키는 것이 최악이다.
3. **JS/템플릿 한계 고지.** JS는 이미 초기화된 페이지에 재주입하는 것이라 fresh load와 다를 수 있다 — 로직이 맞는지까지 확인하고, 최종 확인은 사용자 반영 후 필요하다고 고지. 템플릿 HTML(`{$...}`, 모듈)은 주입으로 시뮬레이션이 불가하므로 이 스킬 범위 밖 — 수정안 코드만 제시하고 "반영 전 검증 불가"를 명시한다.
4. **산출물 전달.** 아래 5개 항목을 **전부** 포함해 보고한다:
   - **파일**: 원본 파일 경로 (스마트디자인 편집창에서 찾을 위치 기준)
   - **원인**: 어떤 규칙/코드가 왜 문제였는지 1~2문장
   - **전체 파일 코드**: 원본 전문에 수정을 반영한 **파일 전체.** snippet·diff·"이 줄만 바꾸세요"가 아니다 — 사용자는 편집창에 통째로 붙여넣는다. 스크래치패드에 파일로 저장하고 경로를 알려주며, 분량이 짧으면 채팅에도 코드블록으로 제시한다.
   - **변경 요약**: 검수용으로 무엇을 어디서 바꿨는지 (여기서만 diff 형태 허용)
   - **반영 방법 + 캐시 주의**: 관리자 → 스마트디자인 편집 → 해당 파일 붙여넣기 저장. 저장 후 옵티마이저 번들 서버 캐시로 잠시 옛 코드가 보일 수 있음을 안내.
5. **종료.** 전달 후 새로고침으로 "최종 확인"하지 않는다 — 주입이 날아가고 배포본(옛 코드)만 렌더되어 판단만 오염된다. 사용자가 반영을 마친 뒤 재검증을 요청하면 그때 번들 본문 확인 + 스크린샷으로 확인한다.

## Common Mistakes (베이스라인 테스트에서 실제 발생)

| 실수 | 결과 |
|---|---|
| 관리자 편집창에서 직접 수정·저장 (또는 제안) | 사용자가 원치 않는 경계 침범. 이 스킬의 종료 지점은 코드 전달이다 |
| 주입 검증을 "임시방편"으로 치부하고 배포로 확인 | 검증 안 된 코드를 사용자가 반복 붙여넣기 — 왕복 낭비 |
| diff/스펙 표/한 줄 교체만 전달 | 사용자가 편집창에서 수동 병합해야 함 → 병합 실수. 반드시 파일 전문 |
| 원본 전문 없이 수정 규칙만 작성 | 4단계에서 전문을 만들 수 없어 다시 역추적 |
| 전달 후 새로고침으로 최종 확인 | 주입 소실 + 옛 코드 렌더 → "반영됐다/깨졌다" 오판 |
